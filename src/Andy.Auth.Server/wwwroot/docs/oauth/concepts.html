OAuth 2.0 & OpenID Connect Concepts

Understand the fundamental concepts behind OAuth 2.0 and OpenID Connect, the industry-standard protocols that power Andy Auth's authentication and authorization.

What is OAuth 2.0?

OAuth 2.0 is an authorization framework that enables applications to obtain limited access to user accounts on third-party services. Instead of sharing passwords, users can grant specific permissions to applications through access tokens.

OAuth vs OpenID Connect

OAuth 2.0 handles authorization (what you can access), while OpenID Connect (OIDC) is built on top of OAuth 2.0 to handle authentication (who you are). Andy Auth implements both protocols.

Key Roles

OAuth 2.0 defines four roles in the authorization process:

Role Description Example
Resource Owner The user who owns the data and grants access You, the end user
Client The application requesting access Claude Desktop, ChatGPT, your web app
Authorization Server Issues access tokens after authentication Andy Auth
Resource Server Hosts protected resources, validates tokens Your API, MCP servers

Grant Types

Andy Auth supports the following OAuth 2.0 grant types:

Authorization Code Grant

The most secure flow for web and mobile applications. Users authenticate directly with Andy Auth and an authorization code is exchanged for tokens.

  • Best for: Web applications, mobile apps, SPAs
  • Security: Tokens never exposed to browser
  • Requires: PKCE for public clients

Client Credentials Grant

Used for server-to-server communication where no user is involved. The client authenticates with its own credentials.

  • Best for: Backend services, scheduled jobs, APIs
  • Security: Requires confidential client with secret
  • Note: No user context in the token

Refresh Token Grant

Allows clients to obtain new access tokens without requiring the user to re-authenticate. Essential for maintaining long-lived sessions.

  • Best for: Any flow requiring extended access
  • Security: Refresh tokens are single-use in Andy Auth
  • Requires: offline_access scope

Token Types

Access Tokens

Short-lived credentials used to access protected resources. Andy Auth issues reference tokens (opaque strings) that must be validated via introspection.

text 
Authorization: Bearer kYjE7d5GS2mLZ8wQP3F1nXrT9vHcA0bU

Refresh Tokens

Long-lived tokens used to obtain new access tokens. They are stored securely and automatically rotated when used.

ID Tokens (OpenID Connect)

JWT tokens containing identity claims about the authenticated user. Used by clients to verify the user's identity.

json 
{
  "iss": "https://auth.example.com/",
  "sub": "user_123456",
  "aud": "my-client-id",
  "exp": 1704067200,
  "iat": 1704063600,
  "name": "John Doe",
  "email": "john@example.com"
}

OpenID Connect

OpenID Connect (OIDC) extends OAuth 2.0 with an identity layer. It provides:

  • ID Tokens: JWT tokens with user identity claims
  • UserInfo Endpoint: API to retrieve user profile data
  • Discovery: Standardized metadata endpoint
  • Standard Claims: Predefined user attributes (name, email, etc.)

Standard Scopes

Scope Description Claims Returned
openid Required for OIDC. Returns ID token sub
profile Basic profile information name, given_name, family_name
email User's email address email, email_verified
offline_access Request a refresh token N/A (returns refresh token)

Security Best Practices

Andy Auth implements modern security best practices:

OAuth 2.1 Requirements

Andy Auth follows OAuth 2.1 guidelines, which means:

  • PKCE is required for all public clients
  • Implicit flow is not supported
  • Resource Owner Password flow is not supported
  • PKCE: Required for all public clients to prevent authorization code interception
  • Token Binding: Tokens are bound to the client that requested them
  • Short Expiration: Access tokens expire after 1 hour by default
  • Token Rotation: Refresh tokens are rotated on each use
  • HTTPS Only: All endpoints require HTTPS in production
