OAuth Client Management

Manage OAuth 2.0 and OpenID Connect client applications. Create new clients, configure redirect URIs, manage secrets, and control client permissions at /admin/clients.

GET /admin/clients

Client Overview

The OAuth Clients page displays all registered client applications in a card-based layout:

OAuth Clients List Screenshot

Each client card shows:

  • Display Name - Human-readable name of the client
  • Client ID - Unique identifier used in OAuth requests
  • Client Type - Public or Confidential
  • DCR Badge - Indicates dynamically registered clients
  • Redirect URIs - Allowed callback URLs
  • Permissions - Granted scopes and grant types

Understanding Client Types

Type Description Use Case
Public Cannot securely store a client secret Single-page apps, mobile apps, CLI tools
Confidential Can securely store a client secret Server-side web applications, backend services
DCR Dynamically registered via RFC 7591 AI assistants, automated client onboarding

Creating a New Client

To create a new OAuth client:

  1. Click the Create Client button
  2. Fill in the client details (see below)
  3. Click Create Client to save
Create Client Form Screenshot

Client Configuration Fields

Field Required Description
Client ID Yes Unique identifier (alphanumeric, hyphens allowed)
Display Name Yes Human-readable name shown to users during consent
Client Type Yes Public or Confidential
Redirect URIs Yes One or more callback URLs (one per line)
Grant Types Yes Authorization code, refresh token, client credentials
Scopes Yes OpenID, profile, email, offline_access

Managing Client Secrets

Important: Save Your Secret

When you create a confidential client, the client secret is displayed only once. Copy and securely store this secret immediately. It cannot be retrieved later.

After creating a confidential client, the secret will be displayed in a highlighted box:

Client Secret Example 
Client Secret: abc123def456ghi789jkl012mno345pqr678stu901vwx234

Regenerating a Client Secret

To generate a new secret for a confidential client:

  1. Find the client in the list
  2. Click Regenerate Secret
  3. Confirm the action in the prompt
  4. Copy the new secret immediately
Secret Rotation

Regenerating a secret invalidates the old one immediately. Make sure to update all applications using this client before regenerating.

Editing a Client

To modify an existing client:

  1. Click the Edit button on the client card
  2. Update the desired fields
  3. Click Save Changes

You can modify:

  • Display name
  • Redirect URIs
  • Grant types
  • Scopes/permissions
Note

The Client ID cannot be changed after creation. If you need a different Client ID, create a new client and delete the old one.

Deleting a Client

To remove an OAuth client:

  1. Click the Delete button on the client card
  2. Confirm the deletion in the prompt
Deletion Warning

Deleting a client is permanent and cannot be undone. All tokens issued to this client will be invalidated, and users will need to re-authenticate with any replacement client.

Managing DCR Clients

Dynamically registered clients (via RFC 7591) have additional management options:

Approving a DCR Client

If your system requires approval for DCR clients:

  1. New DCR clients appear with a "Pending Approval" badge
  2. Click Approve to allow the client to be used

Disabling/Enabling a DCR Client

To temporarily disable a DCR client without deleting it:

  1. Click Disable on the client card
  2. The client will show a "Disabled" badge
  3. Click Enable to restore access

Understanding Permissions

Client permissions are displayed as badges on each card. Common permissions include:

Permission Description
authorization_code Can use the authorization code grant flow
refresh_token Can obtain and use refresh tokens
client_credentials Can authenticate without user involvement
openid Can request OpenID Connect authentication
profile Can access user profile information
email Can access user email address
offline_access Can request refresh tokens for offline use

Best Practices

  • Use descriptive names - Make client names clear and identifiable
  • Minimal permissions - Only grant the scopes and grant types needed
  • Secure redirect URIs - Use HTTPS for all redirect URIs in production
  • Regular audits - Periodically review and remove unused clients
  • Secret rotation - Regenerate secrets periodically for confidential clients
  • Monitor DCR - Review dynamically registered clients regularly
