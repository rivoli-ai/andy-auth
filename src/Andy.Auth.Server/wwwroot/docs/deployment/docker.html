Docker Deployment

Deploy Andy Auth using Docker and docker-compose for containerized deployments with full control over your infrastructure.

When to Use Docker

Docker deployment is ideal for self-hosted environments, on-premises deployments, or when you need full control over the infrastructure. For simpler deployments, consider Railway.

Prerequisites

  • Docker installed (version 20.10+)
  • Docker Compose (version 2.0+)
  • PostgreSQL database (can be containerized or external)

Dockerfile

Andy Auth includes a multi-stage Dockerfile optimized for production:

dockerfile 
# Build stage
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

# Copy csproj and restore
COPY src/Andy.Auth.Server/Andy.Auth.Server.csproj src/Andy.Auth.Server/
RUN dotnet restore src/Andy.Auth.Server/Andy.Auth.Server.csproj

# Copy source and build
COPY src/Andy.Auth.Server/ src/Andy.Auth.Server/
RUN dotnet publish src/Andy.Auth.Server/Andy.Auth.Server.csproj -c Release -o /app/publish

# Runtime stage
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY --from=build /app/publish .

ENTRYPOINT ["dotnet", "Andy.Auth.Server.dll"]

Docker Compose Setup

Create a docker-compose.yml file for a complete deployment with PostgreSQL:

yaml 
version: '3.8'

services:
  andy-auth:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: andy-auth
    ports:
      - "5000:5000"
    environment:
      - ASPNETCORE_ENVIRONMENT=Production
      - ASPNETCORE_URLS=http://+:5000
      - ConnectionStrings__DefaultConnection=Host=postgres;Port=5432;Database=andy_auth;Username=postgres;Password=${POSTGRES_PASSWORD}
      - OpenIddict__UseEphemeralKeys=true
      - CorsOrigins__AllowedOrigins__0=https://myapp.com
      - DynamicClientRegistration__RequireInitialAccessToken=true
      - DynamicClientRegistration__AllowLocalhostRedirectUris=false
    depends_on:
      postgres:
        condition: service_healthy
    restart: unless-stopped
    networks:
      - andy-auth-network

  postgres:
    image: postgres:16-alpine
    container_name: andy-auth-db
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
      - POSTGRES_DB=andy_auth
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped
    networks:
      - andy-auth-network

volumes:
  postgres_data:

networks:
  andy-auth-network:
    driver: bridge

Environment File

Create a .env file to store sensitive configuration:

bash 
# Database
POSTGRES_PASSWORD=your-secure-password-here

# Optional: Azure AD for external authentication
AZUREAD_CLIENT_ID=
AZUREAD_CLIENT_SECRET=
AZUREAD_TENANT_ID=common
Security Warning

Never commit .env files to version control. Add .env to your .gitignore file and use secure secret management in production.

Building and Running

Build the Images

bash 
# Build all services
docker-compose build

# Or build only andy-auth
docker-compose build andy-auth

Start the Services

bash 
# Start in detached mode
docker-compose up -d

# View logs
docker-compose logs -f andy-auth

# Stop services
docker-compose down

# Stop and remove volumes (WARNING: deletes data)
docker-compose down -v

Reverse Proxy with HTTPS

For production, use a reverse proxy like Nginx or Traefik for SSL termination:

Nginx Configuration

nginx 
server {
    listen 443 ssl http2;
    server_name auth.example.com;

    ssl_certificate /etc/letsencrypt/live/auth.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/auth.example.com/privkey.pem;

    # SSL configuration
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers off;

    # Security headers
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "no-referrer" always;

    location / {
        proxy_pass http://localhost:5000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection keep-alive;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}

server {
    listen 80;
    server_name auth.example.com;
    return 301 https://$server_name$request_uri;
}

Docker Compose with Traefik

yaml 
version: '3.8'

services:
  traefik:
    image: traefik:v2.10
    command:
      - "--api.insecure=true"
      - "--providers.docker=true"
      - "--entrypoints.web.address=:80"
      - "--entrypoints.websecure.address=:443"
      - "--certificatesresolvers.letsencrypt.acme.tlschallenge=true"
      - "--certificatesresolvers.letsencrypt.acme.email=admin@example.com"
      - "--certificatesresolvers.letsencrypt.acme.storage=/letsencrypt/acme.json"
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - letsencrypt:/letsencrypt
    networks:
      - andy-auth-network

  andy-auth:
    build: .
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.andy-auth.rule=Host(`auth.example.com`)"
      - "traefik.http.routers.andy-auth.entrypoints=websecure"
      - "traefik.http.routers.andy-auth.tls.certresolver=letsencrypt"
      - "traefik.http.services.andy-auth.loadbalancer.server.port=5000"
    environment:
      - ASPNETCORE_ENVIRONMENT=Production
      - ASPNETCORE_URLS=http://+:5000
      - ConnectionStrings__DefaultConnection=Host=postgres;Port=5432;Database=andy_auth;Username=postgres;Password=${POSTGRES_PASSWORD}
      - OpenIddict__UseEphemeralKeys=true
    depends_on:
      - postgres
    networks:
      - andy-auth-network

  postgres:
    image: postgres:16-alpine
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
      - POSTGRES_DB=andy_auth
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - andy-auth-network

volumes:
  postgres_data:
  letsencrypt:

networks:
  andy-auth-network:

Health Checks

Add health checks to your Docker Compose configuration:

yaml 
andy-auth:
  # ... other configuration ...
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:5000/.well-known/openid-configuration"]
    interval: 30s
    timeout: 10s
    retries: 3
    start_period: 40s

Available health check endpoints:

Endpoint Method Description
/.well-known/openid-configuration GET OpenID Connect discovery - confirms server is operational
/.well-known/jwks GET JSON Web Key Set - confirms signing keys are available

Scaling Considerations

Ephemeral Keys and Scaling

When using ephemeral keys (OpenIddict__UseEphemeralKeys=true), each container instance generates different signing keys. This means tokens issued by one instance cannot be validated by another. For multi-instance deployments, use shared persistent keys.

For horizontal scaling with multiple instances:

  1. Use external PostgreSQL (not containerized) for data persistence
  2. Configure shared signing keys (see Production Checklist)
  3. Use sticky sessions or configure proper load balancing
  4. Consider Redis for distributed session storage

Backup and Restore

Database Backup

bash 
# Backup database
docker-compose exec postgres pg_dump -U postgres andy_auth > backup_$(date +%Y%m%d_%H%M%S).sql

# Restore database
docker-compose exec -T postgres psql -U postgres andy_auth < backup_20240115_120000.sql

Troubleshooting

Container Won't Start

bash 
# Check container logs
docker-compose logs andy-auth

# Check if postgres is healthy
docker-compose ps

# Inspect container
docker inspect andy-auth

Database Connection Issues

  • Ensure PostgreSQL container is running and healthy
  • Verify the connection string uses the Docker service name (postgres) not localhost
  • Check that both containers are on the same Docker network
