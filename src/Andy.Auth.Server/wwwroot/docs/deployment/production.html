Production Deployment Checklist

Security hardening, configuration best practices, and operational guidelines for deploying Andy Auth in production environments.

Security Critical

Andy Auth is an OAuth 2.1 authorization server that handles sensitive authentication data. Proper security configuration is essential. Review this entire checklist before deploying to production.

Pre-Deployment Checklist

Item Status Priority
HTTPS/TLS configured Critical
Database connection secured Critical
Signing keys configured (not ephemeral) Critical
CORS origins restricted Critical
DCR requires initial access tokens Critical
Localhost redirect URIs disabled Critical
Rate limiting configured High
Logging configured (not verbose) High
Backup strategy in place High
Health monitoring configured Medium

Environment Configuration

Required Production Settings

bash 
# Core settings
ASPNETCORE_ENVIRONMENT=Production

# Database - use strong password
ConnectionStrings__DefaultConnection=Host=db.example.com;Port=5432;Database=andy_auth;Username=andy_auth_user;Password=STRONG_PASSWORD;SSL Mode=Require

# For cloud deployments with ephemeral storage (Railway, etc.)
# WARNING: Tokens invalidate on restart
OpenIddict__UseEphemeralKeys=true

# For persistent deployments - use proper certificates
# OpenIddict__Server__SigningKey=BASE64_ENCODED_KEY
# OpenIddict__Server__EncryptionKey=BASE64_ENCODED_KEY

# CORS - specify exact origins, no wildcards
CorsOrigins__AllowedOrigins__0=https://app.example.com
CorsOrigins__AllowedOrigins__1=https://www.example.com

# Dynamic Client Registration - secure settings
DynamicClientRegistration__RequireInitialAccessToken=true
DynamicClientRegistration__AllowLocalhostRedirectUris=false
DynamicClientRegistration__AllowHttpLocalhostRedirectUris=false

# Logging - reduce verbosity
Logging__LogLevel__Default=Warning
Logging__LogLevel__Microsoft.AspNetCore=Warning
Logging__LogLevel__Microsoft.EntityFrameworkCore=Warning

SSL/TLS Configuration

HTTPS Requirements

Andy Auth requires HTTPS in production for:

  • OAuth authorization requests
  • Token endpoint calls
  • All redirect URIs (except localhost in development)
  • OpenID Connect Discovery

Certificate Options

Platform Certificate Source Notes
Railway Automatic (Let's Encrypt) No configuration needed
Azure App Service Azure-managed or custom Free SSL for *.azurewebsites.net
Docker + Nginx Let's Encrypt / Certbot Configure reverse proxy
Docker + Traefik Automatic (ACME) Built-in certificate management

TLS Best Practices

  • Use TLS 1.2 or higher (TLS 1.3 preferred)
  • Disable older protocols (SSL, TLS 1.0, TLS 1.1)
  • Use strong cipher suites
  • Enable HSTS (HTTP Strict Transport Security)

Signing Keys Configuration

Key Persistence

Ephemeral keys (OpenIddict__UseEphemeralKeys=true) regenerate on every restart, invalidating all existing tokens. This is acceptable for UAT but problematic for production where users would need to re-authenticate after every deployment.

Options for Key Management

Option Pros Cons
Ephemeral Keys Simple, no key management Tokens invalidate on restart
Environment Variables Easy configuration Keys visible in environment
Key Vault (Azure/AWS) Most secure, rotation support Requires additional setup
Certificate Files Standard approach File management complexity

Generating Keys

Generate secure 256-bit keys for signing and encryption:

bash 
# Generate random 32-byte (256-bit) keys
openssl rand -base64 32  # For SigningKey
openssl rand -base64 32  # For EncryptionKey

# Or using .NET
dotnet run --project KeyGenerator -- generate-keys

Database Security

Connection String Security

  • Use SSL/TLS for database connections (SSL Mode=Require)
  • Create a dedicated database user with minimal permissions
  • Use strong, unique passwords
  • Never commit connection strings to source control
sql 
-- Create dedicated user with minimal permissions
CREATE USER andy_auth_user WITH PASSWORD 'STRONG_PASSWORD';
GRANT CONNECT ON DATABASE andy_auth TO andy_auth_user;
GRANT USAGE ON SCHEMA public TO andy_auth_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO andy_auth_user;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO andy_auth_user;

Regular Backups

bash 
# Daily backup script
#!/bin/bash
BACKUP_DIR="/backups/andy-auth"
DATE=$(date +%Y%m%d_%H%M%S)
pg_dump -h localhost -U andy_auth_user -d andy_auth | gzip > "$BACKUP_DIR/backup_$DATE.sql.gz"

# Keep only last 30 days
find $BACKUP_DIR -name "backup_*.sql.gz" -mtime +30 -delete

Dynamic Client Registration Security

For production, configure DCR with strict security settings:

json 
{
  "DynamicClientRegistration": {
    "Enabled": true,
    "RequireInitialAccessToken": true,
    "RequireAdminApproval": false,
    "AllowLocalhostRedirectUris": false,
    "AllowHttpLocalhostRedirectUris": false,
    "MaxRedirectUrisPerClient": 5,
    "MaxClientNameLength": 100,
    "AllowedGrantTypes": [
      "authorization_code",
      "refresh_token"
    ],
    "AllowedScopes": [
      "openid",
      "profile",
      "email",
      "offline_access"
    ]
  }
}

Rate Limiting

Andy Auth includes built-in rate limiting. Review and adjust limits for production:

json 
{
  "IpRateLimiting": {
    "EnableEndpointRateLimiting": true,
    "GeneralRules": [
      {
        "Endpoint": "*/Account/Login",
        "Period": "1m",
        "Limit": 10
      },
      {
        "Endpoint": "*/Account/Register",
        "Period": "1h",
        "Limit": 5
      },
      {
        "Endpoint": "*/connect/token",
        "Period": "1m",
        "Limit": 30
      },
      {
        "Endpoint": "POST:*/connect/register",
        "Period": "1h",
        "Limit": 10
      }
    ]
  }
}

Security Headers

Andy Auth automatically adds security headers. Verify they're present in responses:

Header Value Purpose
X-Frame-Options DENY Prevents clickjacking
X-Content-Type-Options nosniff Prevents MIME sniffing
Referrer-Policy no-referrer Controls referrer information
X-Permitted-Cross-Domain-Policies none Restricts Adobe cross-domain
Permissions-Policy restrictive Limits browser features

Health Checks and Monitoring

Health Check Endpoints

Endpoint Method Description
/.well-known/openid-configuration GET OpenID Connect discovery - confirms server is operational
/.well-known/jwks GET JSON Web Key Set - confirms signing keys are available

Monitoring Recommendations

  • Uptime Monitoring - Check /.well-known/openid-configuration every minute
  • Response Time - Alert if token endpoint exceeds 500ms
  • Error Rate - Monitor 4xx/5xx responses on OAuth endpoints
  • Database Health - Monitor connection pool and query times
  • Certificate Expiry - Alert 30 days before SSL certificate expires

Logging Best Practices

Production Logging Configuration

json 
{
  "Logging": {
    "LogLevel": {
      "Default": "Warning",
      "Microsoft.AspNetCore": "Warning",
      "Microsoft.EntityFrameworkCore": "Warning",
      "OpenIddict": "Warning"
    }
  }
}
Sensitive Data Warning

Never log access tokens, refresh tokens, authorization codes, or client secrets. Andy Auth is configured to avoid logging sensitive data, but verify if you add custom logging.

Password Policy

Andy Auth enforces the following password requirements:

  • Minimum 8 characters
  • At least one uppercase letter
  • At least one lowercase letter
  • At least one digit
  • Account lockout after 5 failed attempts (30 minutes)

Recommended Deployment Workflow

  1. Staging Environment - Deploy to staging with production-like configuration
  2. Security Scan - Run security tools (OWASP ZAP, etc.)
  3. Load Testing - Verify performance under expected load
  4. Backup Database - Create backup before production deployment
  5. Deploy to Production - Use blue-green or rolling deployment
  6. Smoke Tests - Verify OAuth flows work correctly
  7. Monitor - Watch logs and metrics for 24 hours
